# DETAIL_DESIGN Review Prompt

Issue ${issue_url} 의 `## Bugfix agent DETAIL_DESIGN` 섹션를리뷰해 주세요.

## 태스크

1. `gh issue view` 로 최신의 Issue 本文를 취득
2. DETAIL_DESIGN 섹션의필수아웃풋를확인
3. IMPLEMENT 스테이트に移행가능か판정

## 완료조건체크리스트

| # | 항목 | 확인내용 |
|---|------|----------|
| 1 | **변경計画** | 대상파일·함수와計画한다변경내용이상세에기재되어 있다 |
| 2 | **구현절차** | 스텝バイ스텝의구현절차이기재되어 있다 |
| 3 | **테스트케이스목록** | 테스트케이스이列挙되어 있다(목적, 입력, 期待결과, Size S/M/L) |
| 4 | **테스트케이스 S/M/L 망라性** | S(単体)·M(연계/이상系)·L(통합/E2E)의 각サイズが少없이とも1건이상존재한다 |
| 5 | **영향문서** | 업데이트이필요한 문서이列挙되어 있다(영향없음의 경우는明示) |
| 6 | **보충정보** | 추가의구현주의事項이 있다(없이ても가능) |

## 금지事項

**次스테이트이후의책무를신규에실행하지 않는다**
- 예：구현, 테스트추가, PR생성

**既에 완료한책무를재실행하지 않는다**
- 예：조사를재실행, 再現테스트를재생성

※ 기재내용의검증(소스코드확인, 정합성체크)는**허가**されてい합니다.

## 출력형식

```markdown
### DETAIL_DESIGN Review Result

#### 검증내용
- <실시한검증와결과를구체적에기재>

#### 체크리스트
- 변경計画(대상/절차): <OK/NG + 구체적근거>
- 구현절차: <OK/NG + 구체적근거>
- 테스트케이스목록: <OK/NG + 구체적근거>
- 테스트케이스 S/M/L 망라性: <OK/NG + 각サイズ의 건数>
- 영향문서: <OK/NG + 구체적근거>
- 보충정보: <OK/NG or N/A>

## VERDICT
- Result: PASS | RETRY
- Reason: <판정이유>
- Evidence: <구체적한 판단근거>
- Suggestion: <RETRY時: 구체적한 설계 수정指示>
```

## 판정가이드ラ인

| 상황 | VERDICT | 다음스테이트 |
|------|---------|-------------|
| 구현에필요한 정보이모두揃い, 모순이 없다 | PASS | IMPLEMENT |
| 부족·모순가 있り설계의見直し이 필요 | RETRY | DETAIL_DESIGN |
| S/M/L いずれか의 테스트サイズが欠如하고 있다 | RETRY | DETAIL_DESIGN |

## レポート방법

1. `gh issue comment` 로 Issue 에 코멘트投稿(VERDICT판정)
2. **PASS판정시만**: 공통규칙에 따라 `gh issue edit` 로 Issue 本文를 업데이트
